3. Thay đổi API contract (additive — không phá cũ)

Proto — thêm field mới, giữ field cũ

Field food_label / food_label_confidence cũ (top-1) giữ nguyên để client cũ vẫn chạy. Thêm repeated:

// inference.proto — thêm message mới
message FoodPrediction {
  string food_label = 1;   // "pho", "com_tam" (đã normalize)
  float  confidence  = 2;  // softmax [0..1]
  float  mass_g      = 3;  // mass theo density của RIÊNG món này
}

// EstimateVolumeResponse: thêm tag kế tiếp (15)
repeated FoodPrediction food_predictions = 15;
Điểm thiết kế quan trọng — mass per-candidate. Volume (depth) là chung cho cả ảnh, nhưng ở chế độ legacy mass = volume × density[label] — density khác nhau theo món. Nên trả mass_g cho từng candidate (volume dùng chung, chỉ tra density theo từng label — chi phí gần như bằng 0). Khi người dùng chọn "phở" thì lấy đúng mass của phở pre-fill. Nếu chạy chế độ primary (depth regressor) thì mass độc lập category → mọi candidate cùng mass, vẫn đúng.

REST — endpoint mới (giữ /estimate cũ làm fallback)

4. Kế hoạch triển khai theo phase

Phase 0

Proto contract dễ · nền tảng

  • Thêm message FoodPrediction + field repeated food_predictions vào AI_server/Computer_Vision/proto/inference.proto
  • Đồng bộ y hệt vào go_backend/api/proto/inference.proto
  • Regen stubs: build_proto.py (Python) + protoc (Go → inference.pb.go)

⚠️ Đây là bước duy nhất "chạm 2 repo cùng lúc". Làm trước, commit, rồi mới làm tiếp.

Phase 1

AI_server — bóc top-K ra dễ

  • ml/worker.py (~L451): đổi topk=1 → topk=5, không bỏ preds[1:]; normalize label từng phần tử (.lower().replace(" ","_"))
  • Tính mass_g per-candidate: volume chung × lookup_density(label_i)
  • domain/entities.py VolumeEstimate: thêm field food_predictions: list[...]
  • grpc_server/service.py: populate field repeated mới
  • Cổng confidence (worker L463): với luồng top-K, trả raw top-5 bất kể ngưỡng — nếu không món đúng nhưng confidence thấp sẽ bị ẩn. Giữ top-1 gate cho backward-compat.
Phase 2

go_backend — phơi candidates + danh sách vừa

  • internal/domain/inference_port.go: InferenceResult thêm Predictions []FoodPrediction
  • infrastructure/grpc_ai_client.go: map field repeated mới; nới validateEstimateResponse (đừng reject <0.5 cho luồng scan)
  • Handler mới POST /foods/scan: gọi EstimateVolume, map mỗi prediction → display_name (Việt hoá từ 30 label), trả list
  • Handler mới GET /foods/by-label: tách logic từ resolveAIFood nhưng trả full list (bỏ selectAIFoodCandidate)
  • Đăng ký route trong NewNutritionHandler
POST /log-meal, struct Food, scaling per-100g — không đụng.
Phase 3

Nutrix — UI nhiều bước vừa · nhiều việc nhất

  • scanner/presentation/scanner_state.dart: thêm state ScannerCandidatesState{ List<FoodCandidate> }
  • scanner_bloc.dart (~L45): handler ảnh gọi /foods/scan → emit candidates thay vì 1 món
  • Bước 2 (mới): widget picker top-5 (bottom sheet) — list category + thanh % confidence. Cần escape hatch "Tìm thủ công" nếu cả 5 đều sai (món ngoài 30 lớp)
  • Bước 3 (mới, tái dùng): chọn category → màn danh sách món. Tái dùng pattern AddMealScreen tab Food: FoodBloc + MealItemCard + ListView.builder, gọi /foods/by-label
  • Bước 4-5 (tái dùng): chọn món → MealDetailScreen (đã có ô chỉnh khối lượng + Add). Pre-fill servingQuantity = mass_g của candidate đã chọn
  • Parse thêm confidence + mass_g trong MealEntity.fromNutrixFood (hiện đang bỏ)
Phase 4

Test & verify dễ

  • gRPC: chụp ảnh phở → kiểm food_predictions trả 5 phần tử sorted
  • REST: POST /foods/scan + GET /foods/by-label?label=pho bằng curl
  • E2E web: chụp → top-5 → chọn phở → danh sách → chọn → chỉnh g → Add → kiểm diary
  • Backward-compat: /foods/estimate cũ vẫn chạy nguyên

6. Vì sao KHÔNG phá hệ thống hiện tại

  • ✓ Proto chỉ thêm field/message mới (tag mới) — gRPC bỏ qua field lạ, client cũ không vỡ
  • ✓ Endpoint /foods/estimate cũ giữ nguyên làm fallback
  • ✓ /log-meal, schema DB, scaling per-100g — không đụng
  • ✓ Cổng confidence top-1 legacy giữ nguyên; chỉ luồng scan mới nới
  • ✓ Không train lại model — softmax/top-K đã có sẵn trong checkpoint
